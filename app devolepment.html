MOBILE APPLICATION DEVELOPMENT

Mobile application development is the process of creating software applications that run on a mobile device, and a typical mobile application utilizes a network connection to work with remote computing resources.

Traditional desktop and laptop PCs are Windows-based with a standard screen size, features and form factor. The mobile landscape is much more fragmented, with four main platforms (Android, iOS, Windows Phone and BlackBerry) that are continually evolving.

Accordingly, building mobile applications requires more factors to consider when planning, including which device platform to build first, what mobile technology option to choose, what capabilities should the mobile phone support, and the need for a long-term roadmap with deployment on additional, next platforms. In contrast to traditional Windows-based desktop development, mobile developers need to think about creating application code for multiple platforms-Android, Windows Phone, iPhone and BlackBerry. Designing an application that is optimized for various devices with different screen densities and screen sizes is even more challenging Applications designed for a desktop or laptop client work with a screen size that far exceeds that of mobile devices. How to design for a device that fits in your pocket requires simplification and a rethink about navigation. While a desktop application may have tabs, options and links on all sides of the screen and in-line with text, for mobile you need to concentrate on the immediate function, reducing the number of options to only those needed for the immediate task or feature.

Devices available from different manufacturers vary from 120 dpi for the lower-end HTC Tattoo / Wildfire and 240 dpi for the higher end Droid series—a difference of 100% in screen density. This means that using hardcoded values for pixels and a single set of images will lead to one of two things on a higher end phone: either your UI will be up-scaled and fuzzy, or the controls will be too small to allow comfortable targeting with a finger. Therefore you need to bundle multiple resolution images and use display metrics to scale your custom drawing code.


Multichannel and Multimodial UIs


We have thus far discussed user interfaces in some detail. We have seen that, depending on the function of the mobile application, we may want to interact with the user through audio, a traditional GUI, or a combination thereof. We have also seen that, because of the wide variety of capabilities among mobile devices and the proliferation of the platforms, there are a great number of user interfaces that we may have to implement for a given application. Although we have looked at mobile GUIs and then VUIs individually, we have not looked at the discipline of building applications that may use more than one channel of communication with the network or interact with the user through more than one mode of their user interface. This is what we will be discussing in this chapter. let us define what multichannel and multimodal mean for these two terms have been incorrectly used widely in the literature on mobile computing. To start, remember that multimodal and multichannel are not the same thing. Multichannel may have two different meetings. First, it can be used to imply a user interface that establishes more than one communication channel to the user. For example, a user interface may present an audio and a video channel to the user.


Intents and Services


A Service is an application component that can perform long-running operations in the background. It does not provide a user interface. Once started, a service might continue running for some time, even after the user switches to another application. Additionally, a component can bind to a service to interact with it and even perform interprocess communication (IPC). For example, a service can handle network transactions, play music, perform file I/O, or interact with a content provider, all from the background.

Wireless Connectivity and Mobile Apps

Technology users and consumers are working with more devices than ever before. Using a smartphone, you can sync a smartwatch, buy a point of sale, control your home alarm, or call a drone. Depending on the device, you may use Bluetooth, Wi-Fi, NFC, or mobile data networks to connect to these devices. As an engineer, you may want to create programs for a specific device, or perhaps upgrade your connected device that can be controlled via a smartphone. Before you uninstall your IDE and start using the hardware, let's take a quick look at Bluetooth, Wi-Fi, NFC and mobile data to learn how this technology works, and which apps are best suited.


NFC

Near Field Communication (NFC) defines a set of protocols that enable two adjacent devices (up to 10 cm apart) to communicate. Like many other proximity card technologies, it uses electromagnetic input where two devices exchange information at a frequency of 13.56MHz. One device can be a weak NFC tag rather than a label or poster, meaning that the reader can simultaneously start reading data and enable tags. Data rates are typically between 106-424 kbit / s. NFC supports three different methods: Card simulation allows the device to function as a smart card. (Actually, this is the method used by Apple Pay and Android Pay. Read / Write mode enables the device to read and write information on NFC tags. Peer mode facilitates the exchange of information between devices in an ad-hoc format. NFC has a variety of programs operating systems, but late payment payment solutions have been the best available for Android and iOS devices. In general, NFC is best suited for applications that require close proximity between the phone and a tag or device and is currently secure, only a limited set of devices that support -NFC. Compatible iOS devices for iPhone 6 / 6S / 6 + / 6S + and AppleWatch. Apple has not yet opened access to the development of these sensors via the API, so it is being used by Apple Pay. -Android offers an API for development, but also, there are not many Android phones on the market including NFC support, especially not for low prices.


Bluetooth

Bluetooth was conceived as a short-term data exchange agreement. It uses the same frequency range as 2.4 GHz Wi-Fi but has 79 channels (less than 4.0 versions), and 40 4.0 and more channels. However, unlike most Wi-Fi applications, it uses a technology called frequency-hopping spread spectrum that constantly changes the channel to avoid congestion caused by competing networks. Data entry rates are also very high, from Bluetooth 1 Mbit / s 1.2 to 24 Mbit / s of 4.0 and above. All iOS and many Android devices support Bluetooth. Some standard Bluetooth applications include wireless headsets and data entry as a modem.


Bluetooth LE

Bluetooth LE (Low Power) is a subset of Bluetooth 4.0 data designed for low power applications. It has a maximum data rate of 0.27 Mbit / s and a power consumption of less than 0.01 W, up to 0.5 W (about half of what a standard Bluetooth device uses.) To put this in perspective: Most powerful smartphone batteries start from at 1400-3000 mAh (milliampere hours) but strives for a day or two of battery life, but it is not uncommon for a 100 LEA-based Bluetooth LE tracking device to last longer for a week or more. Apps that use Bluetooth LE include dynamic trackers, smartwatches, and multiple medical devices.The LE stack contains a few predefined generalized number (GATT) numbers that define communication methods for a variety of devices such as heart rate monitors, pressure gauges. blood vessels, or proximity sensors. This means that if a company was using a tool to track data, it would not need portable software to retrieve data from it. While not as widely available as standard Bluetooth support, Bluetooth LE device support, however, is excellent. All over 4S iPhone models support it, as well as the iPad 3rd generation or newer. On the Android side, things are even more complicated, because the launch of the support that first appeared on Android 4.3 was slow and limited to intermediate mode (meaning communication is blocked on some Bluetooth LE devices) but the phone can't act as a borderline. Android4.4 has provided stable performance, and Android 5.0 has added support for traffic-related advertising. By default, any device that supports Bluetooth 4.0 also supports Bluetooth LE, including older mobile phones such as the Galaxy Nexus and other recent releases in the Nexus range, as well as tablets like the Nexus 7 2013 and newer models. These days, even the most inexpensive Android phones such as the LG Lucky, which sells for about $ 10, also offer Bluetooth 4.0 support


Security

Application security defines security measures at the application level that aim to prevent data or code within the application from being stolen or hijacked. Includes security considerations that occur during application development and build-up, but include systems and ways to protect applications after they have been implemented. Application security may include hardware, software, and processes that identify or minimize security risks. A router that prevents anyone from viewing a computer's IP address on the Internet is a secure application for hardware. But security measures at the application level are also generally built into the software, such as an app firewall that strictly defines what activities are allowed and restricted. Procedures may include things like application security protocols that include protocols such as standard tests. System security is the process of creating, adding, and testing security features within applications to prevent security threats from threats such as unauthorized access and modification.


Why app security is important?

Application security is important because modern applications are often available on various networks and are connected to the cloud, increasing the risk of security threats and breaches. There is increasing pressure and incentive not only to ensure security at the network level but also between applications itself. One reason for this is that hackers are tracking apps with their attacks today more than ever before. Application security tests can reveal vulnerabilities at the application level, helping to prevent these attacks. Developers can also encrypt applications in applications to reduce security risks.

Authentication:

When software developers create processes in an app to ensure that only authorized users have access to it. Authentication procedures ensure that the user is what they claim to be. This can be achieved by requiring the user to provide a username and password when logging in to the application. Multiple authentication requires more than one form of authentication - features can include something you know (password), something you own (mobile), and something you are (thumbprint or face recognition).


Authorization:

After the user is verified, the user can be authorized to access and use the application. The system can verify that the user has permission to access the app by comparing the user's identity with the authorized user list. Authentication must take place before authorization so that the app can only match verified user credentials in the authorized user list.


Encryption:

After the user has verified and used the app, some security measures can prevent sensitive data from being seen or used by the cybercriminal. For cloud-based applications, where traffic containing sensitive data travels between the end user and the cloud, that traffic can be encrypted to keep data safe.


Logging:

If there is a security breach in the application, signing in can help identify who accessed the data and how. App log files provide a timed record of which app features were accessed and by whom.


Application security in the cloud

App security in the cloud brings additional challenges. Because cloud environments provide shared resources, special care should be taken to ensure that users have access to only the data they are authorized to view in their cloud-based applications. Sensitive data is also a major threat to cloud-based applications because that data is transmitted over the Internet from user to app and background.


Mobile application security

Mobile devices also transmit and retrieve information across the Internet, in contrast to the private network, making it vulnerable to attack. Businesses can use virtual private networks (VPNs) to add a layer of mobile app security for employees accessing remote apps. IT departments may also decide to test mobile applications and ensure that they comply with company security policies before allowing employees to use them on mobile devices that connect to the company's network.


What are application security controls?

Application security controls are strategies that enhance the security of the application at the coding level, making it less vulnerable to threats. Many of these controls deal with how the app responds to unexpected inputs that a cybercriminal can use to exploit vulnerabilities. The programmer can write the program code in such a way that the developer can better control the effect of this unexpected input. Fuzzing is a type of app security test where developers test the results of unexpected prices or inputs to find out what makes the system work in an unexpected way that could open a security hole.
